MyCAE 轻量级 CAE 集成系统说明文档

## 1. 概述

### 1.1 项目背景

CAE 是 Computer Aided Engineering 的缩写，中文通常称为“计算机辅助工程分析”。它的作用是利用计算机对工程结构进行仿真分析，例如计算零件受力后的变形、应力分布、支反力以及危险区域。对于机械、土木、航空、汽车等工程领域来说，CAE 软件可以在真实制造之前提前判断设计是否合理。

传统 CAE 软件功能很强，例如 Abaqus、ANSYS、HyperMesh 等，但它们通常存在学习成本高、软件流程复杂、内部数据流不直观等问题。对于初学者来说，常常不清楚“几何、网格、材料、边界条件、载荷、求解、后处理”之间到底是什么关系。

MyCAE 项目面向轻量级 CAE 学习和原型开发场景，目标是把一个结构仿真的核心流程集成到一个 Qt 桌面软件中，让用户能够通过界面完成从几何建模到结果显示的完整过程。

### 1.2 项目简介

MyCAE 是一个基于 Qt/C++ 开发的轻量级 CAE 集成系统。当前版本重点实现结构静力有限元分析流程，集成了 Open CASCADE、Gmsh、CalculiX 和 VTK 等开源组件。

项目当前主流程如下：

```text
创建工程
-> 创建或导入几何
-> 选择几何面并创建面组
-> 设置面组为物理组
-> 生成有限元网格
-> 创建材料
-> 创建材料分区
-> 创建边界条件
-> 创建载荷
-> 调用 CalculiX 求解
-> 读取结果并显示云图
```

通俗地说，本项目希望实现：

```text
在一个软件里完成结构仿真的建模、网格、设置、求解和结果查看。
```

### 1.3 项目目标

本项目的主要目标包括：

1. 建立一个 Qt 桌面 CAE 软件基础框架。
2. 实现基础三维几何创建和 STEP 几何导入。
3. 实现几何显示、几何边线显示和面选择功能。
4. 实现面组管理，并将面组转换为 Gmsh 物理组。
5. 调用 Gmsh 生成三维四面体网格。
6. 支持网格显示、网格边线显示和网格透明显示。
7. 支持结构材料、材料分区、边界条件和载荷定义。
8. 调用 CalculiX 完成结构静力分析。
9. 通过 VTK 显示位移、应力、反力等结果场。
10. 为后续扩展 OpenFOAM、AI Agent 自动建模、多求解器调度提供架构基础。

### 1.4 项目定位

MyCAE 当前更适合定位为：

```text
轻量级 CAE 教学与原型系统
```

它不是成熟商业 CAE 软件的完整替代品，而是一个能够展示 CAE 核心流程的软件原型。它的价值在于把复杂 CAE 流程拆成清晰模块，让初学者可以理解每一步做了什么，也方便后续继续扩展。

### 1.5 基础概念说明

为了让没有 CAE 基础的读者也能看懂，下面先解释几个核心概念。

| 概念 | 简单解释 | 在 MyCAE 中的作用 |
|---|---|---|
| 几何 | 要分析的三维模型，例如长方体、圆柱体、STEP 零件 | 由 Open CASCADE 创建和管理 |
| 网格 | 把连续几何切分成很多小单元 | 由 Gmsh 生成，供有限元求解使用 |
| 节点 | 网格上的点 | 求解器计算位移的位置 |
| 单元 | 由节点组成的小体块，例如四面体 | 求解器计算应力和刚度的基本对象 |
| 面组 | 把几何上的一个或多个面保存成一个集合 | 用于固定、加载、设置物理组 |
| 物理组 | Gmsh 中带有物理意义的几何区域 | 网格生成后保留边界身份 |
| 材料 | 描述结构的物理属性，例如弹性模量、泊松比 | 告诉求解器结构是什么材料 |
| 材料分区 | 把材料分配给某一组实体单元 | 当前常用 `EALL` 表示全部实体单元 |
| 边界条件 | 限制结构运动，例如固定某个面 | 防止模型发生刚体运动 |
| 载荷 | 施加在结构上的外力，例如集中力、压力、重力 | 用于模拟受力情况 |
| 求解器 | 真正进行有限元计算的程序 | 当前主要使用 CalculiX |
| 后处理 | 查看求解结果，例如位移云图、应力云图 | 当前主要使用 VTK 显示 |

## 2. 功能

### 2.1 功能总览

当前 MyCAE 已经具备以下主要功能：

| 功能模块 | 已实现功能 |
|---|---|
| 工程管理 | 新建工程、打开工程、工程树展示、工程数据保存 |
| 几何建模 | 创建长方体、圆柱体、球体，导入 STEP 几何 |
| 几何显示 | 显示几何实体、显示几何边线、支持基础视图交互 |
| 面选择 | 在三维窗口中选择几何面 |
| 面组管理 | 创建面组、编辑面组、删除面组、设置面组为物理组 |
| 网格划分 | 调用 Gmsh 生成三维四面体网格 |
| 网格显示 | 显示网格、显示网格边线、支持网格透明显示 |
| 材料管理 | 创建结构材料，支持弹性模量、泊松比等参数 |
| 材料分区 | 将材料分配给实体单元集合，例如 `EALL` |
| 边界条件 | 支持固定约束、位移约束、载荷目标等 |
| 载荷定义 | 支持集中力、表面力、压力、重力等载荷类型 |
| CalculiX 求解 | 生成 `.inp` 文件、调用求解器、读取结果 |
| 结果后处理 | 显示位移、应力、反力等结果场 |
| 结果导出 | 支持 CSV、报告、截图等导出能力 |
| 插件扩展 | 已有 CalculiX 和 OpenFOAM 插件注册机制，支持内置插件和外部进程插件扩展 |
| OpenFOAM 微服务 | 通过 HTTP/JSON 调用本地 FastAPI 服务，预留 CFD 求解远程执行能力 |

### 2.2 工程管理功能

工程管理模块用于组织一个仿真项目的所有数据。一个工程通常包含几何文件、网格文件、求解文件、结果文件和日志文件。

工程目录中常见内容如下：

```text
project.json       工程基础信息
geometry/          几何数据文件
mesh/              网格数据文件
solver/            求解器输入输出文件
results/           结果文件
logs/              日志文件
```

用户可以通过工程树查看当前工程中的几何、网格、材料、边界条件、载荷和结果对象。

### 2.3 几何建模功能

几何模块用于创建或导入仿真对象的三维形状。当前支持：

1. 创建长方体。
2. 创建圆柱体。
3. 创建球体。
4. 导入 STEP 几何文件。
5. 显示几何实体。
6. 显示几何边线。
7. 对几何进行基础变换。

例如，在悬臂梁算例中，可以创建一个长方体作为梁：

```text
长度：200 mm
宽度：40 mm
高度：40 mm
```

几何显示负责把模型显示到三维窗口中；几何边线显示负责叠加模型轮廓，方便用户观察端面并进行选面。

### 2.4 面组与物理组功能

面组是本项目中非常重要的概念。它的作用是把用户选择的几何面保存成一个有名字的集合。

例如悬臂梁分析中：

```text
左端面 -> Fixed_End
右端面 -> Load_End
```

`Fixed_End` 后续用于固定约束，`Load_End` 后续用于施加载荷。

设置面组为物理组的含义是：

```text
告诉 Gmsh：这个面在网格生成后仍然需要保留边界身份。
```

如果不设置物理组，网格生成后可能无法准确知道哪些节点和单元面属于固定端或加载端。

### 2.5 网格划分功能

网格模块负责把几何模型转化为有限元求解需要的节点和单元。当前项目通过 Gmsh 生成三维四面体网格。

典型流程如下：

```text
STEP / BREP 几何
-> Gmsh 生成 .msh 文件
-> MshReader 读取节点和单元
-> MeshData 保存网格数据
-> VTK 显示网格
```

网格参数包括：

| 参数 | 含义 |
|---|---|
| 最小网格尺寸 | 控制最小单元大小 |
| 最大网格尺寸 | 控制最大单元大小 |
| Tetra4 | 一阶四面体单元，适合演示 |
| Tetra10 | 二阶四面体单元，精度更高但更复杂 |

支持“网格透明显示”。该功能可以通过菜单切换：

```text
仿真 -> 网格透明显示
```

开启后，网格以半透明方式显示，方便观察模型内部四面体单元。该功能只影响显示效果，不改变网格数据，也不影响求解结果。

### 2.6 材料与材料分区功能

材料用于描述结构的物理属性。结构静力分析中最常用的材料参数包括：

| 参数 | 含义 | 示例 |
|---|---|---|
| 弹性模量 E | 材料抵抗变形的能力 | 钢材常用 `210000 MPa` |
| 泊松比 nu | 横向变形与纵向变形的比例 | 钢材常用 `0.30` |
| 密度 | 单位体积质量 | 重力分析时更重要 |

材料分区用于指定：

```text
哪些实体单元使用哪种材料。
```

当前演示中最常用的单元集是：

```text
EALL
```

它表示全部实体单元。对于只有一个实体区域的模型，通常选择 `EALL` 即可。界面中也可能出现 `MyCAE_Domain` 或 `tag_1`，它们来自 Gmsh 的三维物理体区域。单实体模型中它们通常和全部实体区域接近。

### 2.7 边界条件功能

边界条件用于限制模型运动，防止结构出现刚体运动。

当前结构边界条件主要包括：

| 类型 | 作用 |
|---|---|
| FixedSupport | 固定支撑，使目标区域位移为 0 |
| Displacement | 指定位移约束 |
| LoadTarget | 声明某个边界区域可以作为载荷目标 |
| Wall | 更偏向 CFD 或边界语义预留 |

在悬臂梁演示中，左端面通常设置为固定约束：

```text
Ux = 0
Uy = 0
Uz = 0
```

这表示左端面在 X、Y、Z 三个方向都不能移动。

### 2.8 载荷功能

载荷用于模拟外部作用力。当前支持的载荷类型包括：

| 载荷类型 | 含义 | 说明 |
|---|---|---|
| Force | 集中力或等效总力 | 使用 X/Y/Z 三个方向分量 |
| SurfaceForce | 表面总力 | 当前按表面节点等效处理 |
| Pressure | 压力 | 标量，表示单位面积上的力 |
| Gravity | 重力 | 对整体施加重力场 |
| BodyForce | 体力数据结构 | 扩展预留 |
| Temperature | 温度载荷数据结构 | 扩展预留 |

Force 的 X/Y/Z 表示力的方向分量。例如：

```text
X = 0
Y = 0
Z = -1000
```

表示施加一个沿 Z 负方向的 1000 N 力。

Pressure 是标量，不是向量。当前界面中压力可能只显示一个数值输入框，并复用 `X` 字段存储压力大小。这里的 `X` 不是 X 方向压力，而是压力大小。

当前项目还不支持直接点选单个节点施加载荷。当前载荷主要作用在面组或边界目标上。如果要对多个面施加载荷，正确方式是先把多个面加入同一个面组，再把这个面组作为载荷目标。

### 2.9 求解功能

当前项目主要调用 CalculiX 进行结构静力分析。

求解流程如下：

```text
工程数据
-> SimulationCase
-> CalculiX 输入文件 .inp
-> 调用 ccx.exe
-> 生成 .dat / .frd / .sta 等文件
-> 读取结果
-> 显示云图
```

求解前必须具备：

1. 有几何。
2. 有网格。
3. 有结构材料。
4. 有材料分区。
5. 有边界条件。
6. 有载荷。
7. CalculiX 可执行文件可用。

### 2.10 结果后处理功能

后处理用于查看求解结果。当前结果场包括：

| 结果场 | 含义 | 用途 |
|---|---|---|
| `Ux` | X 方向位移 | 查看结构沿 X 方向移动多少 |
| `Uy` | Y 方向位移 | 查看结构沿 Y 方向移动多少 |
| `Uz` | Z 方向位移 | 查看结构沿 Z 方向移动多少 |
| `Displacement Magnitude` | 总位移大小 | 查看整体变形趋势 |
| `Von Mises Stress` | 等效应力 | 判断危险区域 |
| `RF1` | X 方向反力 | 查看 X 方向支反力 |
| `RF2` | Y 方向反力 | 查看 Y 方向支反力 |
| `RF3` | Z 方向反力 | 查看 Z 方向支反力 |
| `Reaction Force Magnitude` | 反力合量 | 查看总支反力 |

其中 `RF1/RF2/RF3` 是 CalculiX 中的反力结果：

```text
1 = X 方向
2 = Y 方向
3 = Z 方向
```

如果悬臂梁右端受到向下的 `Z = -1000 N` 载荷，固定端通常会出现相反方向的 `RF3` 反力。

## 3. 架构与程序设计

### 3.1 总体架构

MyCAE 采用分层架构设计，将用户界面、流程控制、领域数据、第三方库适配和文件读写分离。

总体结构如下：

```text
Qt 用户界面层
-> Workflow / Controller 流程控制层
-> ProjectModel / Repository 数据管理层
-> OCC / Gmsh / CalculiX / VTK 适配层
-> 文件系统与外部求解器
```

这种设计的好处是模块职责清晰。界面只负责交互，数据层负责保存工程对象，适配层负责对接第三方库，求解器模块负责生成输入文件和读取结果。

### 3.2 模块划分

项目主要模块如下：

| 模块 | 主要职责 |
|---|---|
| `ui` | 主窗口、菜单、对话框、工程树、属性面板、三维显示 |
| `geometry` | 几何对象、几何元数据、几何管理 |
| `occ` | Open CASCADE 几何创建、读写和适配 |
| `mesh` | 网格数据结构、Gmsh 调用、MSH 读取、边界恢复 |
| `solver` | 材料、边界条件、载荷、结构工况、求解器数据 |
| `solver/calculix` | CalculiX 输入文件生成、求解调用、结果读取 |
| `result` | 结果对象、结果场、极值计算、导出 |
| `workflow` | 高层流程组织，例如求解前检查、工程流程控制 |
| `commands` | 菜单命令和显示命令 |
| `validation` | 数据校验和预检查 |

### 3.3 核心数据对象

系统中的核心对象包括：

| 数据对象 | 含义 |
|---|---|
| `ProjectModel` | 当前工程的总数据模型 |
| `GeometryObject` | 几何对象 |
| `MeshObject` | 网格对象 |
| `MeshData` | 节点、单元、物理组等网格数据 |
| `Material` | 材料数据 |
| `SectionAssignment` | 材料分区 |
| `BoundaryCondition` | 边界条件 |
| `Load` | 载荷 |
| `SimulationCase` | 求解工况 |
| `ResultObject` | 求解结果对象 |

### 3.4 数据流设计

一次完整结构求解的数据流如下：

```text
用户创建几何
-> GeometryObject 保存几何信息
-> OCC 写出 STEP / BREP 文件
-> Gmsh 读取几何并生成 .msh 网格
-> MshReader 读取节点、单元和物理组
-> MeshData 保存网格数据
-> 用户创建材料、材料分区、边界条件和载荷
-> SimulationCaseBuilder 组装求解工况
-> CalculiXInputDeckBuilder 生成输入文件
-> CalculiXRunner 调用 ccx.exe
-> CalculiXResultReader 读取结果
-> VTK 显示结果云图
```

### 3.5 UI 与业务逻辑关系

用户在界面中的操作会被转换为工程数据对象。

例如：

| 用户操作 | 程序内部变化 |
|---|---|
| 创建长方体 | 新增 `GeometryObject` |
| 创建面组 | 保存面 ID 和面组名称 |
| 生成网格 | 新增 `MeshObject` 和 `MeshData` |
| 创建材料 | 新增 `Material` |
| 创建材料分区 | 新增 `SectionAssignment` |
| 创建固定约束 | 新增 `BoundaryCondition` |
| 创建载荷 | 新增 `Load` |
| 运行求解 | 新增 `ResultObject` |

这种设计让用户操作和工程数据保持一致，也方便保存、重新打开和后续扩展。

### 3.6 求解器插件化设计

MyCAE 的求解器不是直接写死在主窗口中的，而是通过插件接口统一管理。这样做的目的是让不同求解器都遵守同一套调用流程，主程序不需要关心某个求解器内部到底是 CalculiX、OpenFOAM，还是后续新增的其他求解器。

求解器插件的核心接口可以概括为三步：

```text
exportCase   导出求解器输入文件或请求文件
runCase      执行求解过程
readResult   读取求解结果并返回给 MyCAE
```

也就是说，不管是哪一种求解器插件，都需要回答三个问题：

1. 如何把 MyCAE 的工程数据转换成该求解器能理解的输入？
2. 如何启动该求解器执行计算？
3. 如何把求解结果重新读回 MyCAE？

当前项目中，求解器插件接口主要包含：

| 接口 | 作用 |
|---|---|
| `descriptor()` | 返回插件名称、ID、版本、求解类型和能力描述 |
| `exportCase()` | 导出求解工况 |
| `runCase()` | 运行求解器 |
| `readResult()` | 读取结果文件 |

求解器插件管理器负责注册和查找插件。当前内置插件包括：

| 插件 ID | 插件名称 | 求解类型 | 当前状态 |
|---|---|---|---|
| `calculix` | CalculiX | 结构有限元分析 | 可用于主线演示 |
| `openfoam` | OpenFOAM | CFD 流体分析 | 微服务演示集成 |

插件化设计的优点是：

1. 主程序只需要调用统一接口，不需要为每个求解器写一套界面逻辑。
2. CalculiX 和 OpenFOAM 可以采用不同执行方式，但对上层流程保持一致。
3. 后续可以继续增加外部进程插件，例如自定义 Python 求解脚本。
4. 每个插件可以声明自己的能力，例如是否支持导出、运行、读取结果。
5. 便于后续实现多求解器切换和自动化调度。

### 3.7 CalculiX 插件设计

CalculiX 是当前结构静力分析主线使用的求解器插件。

CalculiX 插件负责：

1. 从工程数据中提取结构分析所需信息。
2. 检查网格、材料、材料分区、边界条件和载荷是否完整。
3. 生成 CalculiX 输入文件 `.inp`。
4. 调用外部 `ccx.exe` 执行求解。
5. 读取 `.dat`、`.frd`、`.sta` 和日志文件。
6. 将位移、应力和反力结果交给 VTK 后处理模块显示。

CalculiX 插件的执行流程如下：

```text
ProjectModel
-> SimulationCase
-> SolverCaseContext
-> CalculiXPlugin.exportCase()
-> CalculiXPlugin.runCase()
-> CalculiXPlugin.readResult()
-> ResultObject
-> VTK 后处理显示
```

对于用户来说，点击“运行 CalculiX”时看到的是一个按钮操作；对于程序来说，实际执行的是插件接口中的导出、运行和读取三个阶段。

### 3.8 OpenFOAM 微服务设计

OpenFOAM 的运行环境通常比 CalculiX 更复杂，它依赖 Linux 环境、OpenFOAM case 目录结构、求解器命令、网格转换和后处理文件。为了避免把这些复杂环境全部塞进 Qt 桌面程序，本项目采用了“微服务式”的集成思路。

当前 OpenFOAM 插件不是直接在 Qt 程序内部运行完整 OpenFOAM，而是通过 HTTP/JSON 请求调用一个本地或远程服务。该服务可以用 FastAPI 实现，负责处理 OpenFOAM 风格的 case 生成、运行和结果返回。

OpenFOAM 微服务架构如下：

```text
MyCAE 桌面端
-> OpenFoamPlugin
-> OpenFoamCaseWriter 导出请求清单
-> OpenFoamServiceClient 发送 HTTP POST /run
-> FastAPI OpenFOAM 服务
-> 生成或运行 OpenFOAM case
-> 输出 VTK / 日志 / JSON 响应
-> MyCAE 读取响应并导入结果
```

该设计中，各部分职责如下：

| 模块 | 职责 |
|---|---|
| MyCAE Qt 客户端 | 管理工程、几何、网格、边界、材料和结果显示 |
| `OpenFoamPlugin` | 作为 MyCAE 内部的 OpenFOAM 求解器插件 |
| `OpenFoamCaseWriter` | 导出 OpenFOAM 请求清单 `openfoam_case_request.json` |
| `OpenFoamServiceClient` | 向服务端发送 HTTP 请求并保存响应 |
| FastAPI 服务端 | 接收请求、组织 OpenFOAM case、执行求解或演示任务 |
| OpenFOAM 环境 | 实际运行 CFD 求解器 |
| VTK 后处理 | 将服务返回的 VTK 文件显示到 MyCAE 中 |

当前 OpenFOAM 服务默认地址为：

```text
http://127.0.0.1:8765
```

运行请求接口为：

```text
POST /run
```

请求数据主要包含：

| 字段 | 含义 |
|---|---|
| `caseName` | 工况名称 |
| `caseDirectory` | 求解工况目录 |
| `projectName` | 当前工程名称 |
| `meshName` | 当前 CFD 网格名称 |
| `meshNodeCount` | 网格节点数量 |
| `meshCellCount` | 网格单元数量 |
| `materialCount` | CFD 材料数量 |
| `boundaryConditionCount` | CFD 边界数量 |
| `loadCount` | CFD 场值或载荷数量 |

服务端返回结果主要包含：

| 字段 | 含义 |
|---|---|
| `success` | 服务端任务是否成功 |
| `jobId` | 服务端任务编号 |
| `summary` | 求解或演示摘要 |
| `logFile` | 服务日志文件 |
| `vtkFiles` | 可导入 MyCAE 后处理的 VTK 文件 |
| `log` | 服务端返回的日志消息 |

这种微服务设计的优点是：

1. Qt 桌面程序不需要直接绑定复杂的 OpenFOAM 运行环境。
2. OpenFOAM 可以部署在本机、WSL、Linux 服务器或远程计算节点上。
3. CFD 求解流程可以独立升级，不影响桌面端主程序。
4. 后续可以接入 AI Agent，让 Agent 在服务端自动生成 OpenFOAM case。
5. 服务端可以扩展队列、任务 ID、远程日志、异步求解和结果下载。
6. 结构求解和流体求解在 MyCAE 中仍然保持统一的插件调用方式。

## 4. 实施方案

### 4.1 总体实施思路

本项目采用“先打通最小 CAE 闭环，再逐步扩展功能”的实施方式。

整体实施路线为：

```text
项目骨架
-> Qt 主界面
-> 几何创建
-> 几何显示
-> 面选择与面组
-> Gmsh 网格
-> 材料与工况
-> CalculiX 求解
-> VTK 后处理
-> 导出与测试
```

### 4.2 第一阶段：工程框架与主界面

第一阶段实现 Qt 桌面程序基础框架，包括主窗口、菜单栏、工具栏、工程树、属性面板、日志面板和三维显示区域。

目标是让软件具备基本交互入口：

```text
用户可以启动软件
-> 创建工程
-> 在工程树中查看对象
-> 在三维窗口中显示模型
```

### 4.3 第二阶段：几何建模与显示

第二阶段集成 Open CASCADE，实现基础几何创建和 STEP 导入。

主要工作包括：

1. 创建长方体、圆柱体、球体。
2. 保存几何元数据。
3. 导出 STEP / BREP 文件。
4. 在 VTK 窗口中显示几何。
5. 支持几何边线显示。

该阶段使系统从普通界面程序变成具备三维几何能力的软件。

### 4.4 第三阶段：面组与物理组

第三阶段实现面选择和面组管理。

主要工作包括：

1. 在三维视图中选择几何面。
2. 将选择结果保存为面组。
3. 支持面组命名、编辑和删除。
4. 支持将面组设置为 Gmsh Physical Surface。
5. 让面组可以被边界条件和载荷引用。

该阶段解决了“用户选择的几何面如何传递到网格和求解阶段”的问题。

### 4.5 第四阶段：网格划分

第四阶段集成 Gmsh，实现三维网格生成。

主要工作包括：

1. 根据几何文件生成 Gmsh 输入。
2. 调用 `gmsh.exe`。
3. 读取 `.msh` 文件。
4. 解析节点、四面体单元、表面三角形和物理组。
5. 转换为 VTK 网格进行显示。
6. 增加网格透明显示功能。

网格阶段完成后，系统就可以将连续几何转化为有限元可计算的离散模型。

### 4.6 第五阶段：材料、约束和载荷

第五阶段实现结构分析工况定义。

主要工作包括：

1. 创建结构材料。
2. 创建材料分区。
3. 将材料分配给 `EALL` 或指定三维物理体单元集。
4. 创建固定约束和位移约束。
5. 创建载荷目标。
6. 创建集中力、表面力、压力和重力。

| 类别 | 推荐设置 |
|---|---|
| 几何 | 长 200 mm，宽 40 mm，高 40 mm |
| 材料 | Steel / 铁 |
| 弹性模量 | 210000 MPa |
| 泊松比 | 0.30 |
| 单元集 | `EALL` |
| 固定端 | `Fixed_End` |
| 加载端 | `Load_End` |
| 集中力 | `Z = -1000 N` |

### 4.7 第六阶段：CalculiX 求解

第六阶段实现 CalculiX 输入文件生成和求解调用。

主要工作包括：

1. 写出节点 `*NODE`。
2. 写出单元 `*ELEMENT`。
3. 写出材料 `*MATERIAL`。
4. 写出材料分区 `*SOLID SECTION`。
5. 写出约束 `*BOUNDARY`。
6. 写出载荷 `*CLOAD` 或 `*DLOAD`。
7. 调用 `ccx.exe`。
8. 读取 `.dat`、`.frd`、`.sta` 等结果。

求解前系统会进行预检查。如果材料分区绑定到旧几何或旧网格，会提示：

```text
CalculiX section assignment targets another mesh or geometry
```

此时需要删除旧材料分区，重新为当前几何和当前网格创建 `EALL` 材料分区。

### 4.8 第七阶段：结果后处理

第七阶段实现结果场显示和导出。

主要工作包括：

1. 读取节点位移。
2. 读取节点反力。
3. 读取单元应力。
4. 计算位移模。
5. 计算 Von Mises 应力。
6. 生成 VTK 结果网格。
7. 显示云图。
8. 显示最大值和最小值。
9. 支持 CSV、报告和截图导出。

## 6. 关键技术（亮点）

### 6.1 多开源组件集成

项目集成了多个典型 CAE 开源组件：

```text
Qt + Open CASCADE + Gmsh + CalculiX + VTK
```

这些组件分别负责界面、几何、网格、求解和可视化。项目通过 C++ 代码将它们连接起来，形成完整工作流。

### 6.2 完整 CAE 闭环

项目最核心的亮点是打通了完整 CAE 流程：

```text
几何
-> 网格
-> 材料
-> 约束
-> 载荷
-> 求解
-> 后处理
```

这说明项目不是单独实现某一个界面功能，而是具备轻量级 CAE 软件的基本形态。

### 6.3 面组与物理组机制

面组和物理组机制解决了几何面到求解边界的映射问题。

用户在几何上选择一个面并创建面组后，系统可以把该面组写入 Gmsh 物理组。网格生成后，物理组信息可以被恢复为边界节点和边界面，从而用于固定约束、压力载荷等。

这一机制连接了：

```text
几何选择
-> Gmsh 网格边界
-> CalculiX 边界条件和载荷
```

### 6.4 材料分区与实体单元集

材料分区功能用于将材料分配到实体单元集合。当前最常用的是 `EALL`，表示全部实体单元。

该功能对应 CalculiX 中的：

```text
*SOLID SECTION
```

通过材料分区，系统可以告诉求解器每个实体单元使用哪种材料。

### 6.5 CalculiX 输入文件自动生成

系统可以根据用户在界面中创建的材料、边界条件和载荷，自动生成 CalculiX 输入文件。用户不需要手动编写 `.inp` 文件。

自动生成内容包括：

1. 节点。
2. 单元。
3. 材料。
4. 单元集。
5. 表面集。
6. 约束。
7. 载荷。
8. 输出请求。

### 6.6 结果场可视化

项目不仅可以读取位移和应力，还可以显示反力结果。

当前支持：

```text
Ux / Uy / Uz
Displacement Magnitude
Von Mises Stress
RF1 / RF2 / RF3
Reaction Force Magnitude
```

这让用户可以从多个角度观察结构响应。

### 6.7 网格透明显示

该功能可通过菜单切换网格显示透明度，便于观察内部四面体单元。

```text
有限元网格不是只有表面，而是整个实体内部都被划分成单元。
```

### 6.8 可扩展架构

项目采用模块化设计，后续可以继续扩展：

1. 点载荷。
2. 多材料分区。
3. 多体几何。
4. OpenFOAM CFD 求解。
5. AI Agent 自动创建仿真工况。
6. 自动生成仿真报告。

### 6.9 求解器插件化亮点

求解器插件化是本项目架构上的重要亮点。传统写法可能会把“运行 CalculiX”直接写在某个按钮事件中，这样后续如果要接入 OpenFOAM 或其他求解器，就需要修改大量界面代码。

MyCAE 采用统一插件接口，把求解器能力抽象为：

```text
导出工况 -> 运行求解 -> 读取结果
```

这样 CalculiX 和 OpenFOAM 虽然内部实现完全不同，但在主程序看来都是一个 `SolverPlugin`。这使得软件具备继续扩展多求解器的基础。

### 6.10 OpenFOAM 微服务亮点

OpenFOAM 微服务设计体现了桌面端与计算端解耦的思想。Qt 桌面程序负责工程管理、参数组织和结果显示，OpenFOAM 服务端负责 CFD case 生成、运行和结果文件输出。

这种设计相比直接在 Qt 中调用 OpenFOAM 命令有几个优势：

1. OpenFOAM 环境可以独立部署，不污染 Windows 桌面程序环境。
2. 服务端可以放在 WSL、Linux 服务器或远程计算节点上。
3. Qt 端和服务端之间只通过 JSON 通信，接口清晰。
4. 后续可以扩展异步任务队列、远程求解、日志监控和结果下载。
5. AI Agent 可以接入服务端，自动生成或修改 OpenFOAM case。

因此，OpenFOAM 微服务不是单纯“调用另一个程序”，而是为后续 CFD 自动化和远程仿真平台化预留了清晰接口。

### 7.3 OpenFOAM 微服务运行方式

```text
http://127.0.0.1:8765
```

服务启动命令示例：

```powershell
python -m uvicorn openfoam_demo_service:app --host 127.0.0.1 --port 8765
```

启动服务后，MyCAE 中的 OpenFOAM 插件会向下面的接口发送请求：

```text
POST http://127.0.0.1:8765/run
```

如果需要修改服务地址，可以设置环境变量：

```powershell
$env:MYCAE_OPENFOAM_SERVICE_URL="http://127.0.0.1:8765"
```

OpenFOAM 微服务运行后，MyCAE 会在求解工况目录下保存请求和响应文件：

```text
openfoam_case_request.json      MyCAE 发给服务端的请求清单
openfoam_service_result.json    服务端返回的结果 JSON
openfoam_demo.log               服务端日志文件
```

如果服务没有启动，OpenFOAM 插件会提示请求失败或超时。此时应先确认：

1. FastAPI 服务是否已经启动。
2. 服务端口是否为 `8765`。
3. `MYCAE_OPENFOAM_SERVICE_URL` 是否配置正确。
4. 防火墙或代理是否阻止了本地 HTTP 请求。
