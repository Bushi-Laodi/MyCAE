# MyCAE 三人并行开发接口契约

当前阶段目标：

```text
主要接入 CalculiX；
OpenFOAM 只保留同构接口和 placeholder；
三个人围绕标准 SolverPlugin 机制并行开发。
```

本文档用于保证三个人能同时开工，并避免抢同一批公共文件。

## 1. 当前是否可以并行

可以。

当前已经预留了并行开发需要的公共接口：

```text
MeshBoundary
ResultObject
ResultRepository
SolverCaseWriterResult
SolverCaseWriter
OpenFoamCaseWriter placeholder
```

并且 `ProjectContext` 已经具备：

```text
ProjectContext
├─ GeometryRepository
├─ MeshRepository
├─ SolverRepository
├─ ResultRepository
└─ SelectionState
```

后续三个人主要填自己的模块，不需要一开始同时改 `ProjectContext` 或 `ProjectModel`。

## 2. 三人并行主线

### 成员 A：插件架构与集成

负责：

```text
标准 SolverPlugin 接口
CalculiXPlugin 集成
OpenFOAM placeholder
SolverPluginManager
SolverCaseWorkflowController
```

主要文件：

```text
src/solver/plugin/SolverPlugin.h
src/solver/plugin/SolverPluginManager.*
src/solver/export/SolverCaseWriter.*
src/solver/export/SolverCaseWriterResult.h
src/solver/export/OpenFoamCaseWriter.*
src/solver/calculix/CalculiXPlugin.*
src/workflow/SolverCaseWorkflowController.*
```

成员 A 的原则：

```text
定义接口；
串联 B/C 的 CalculiXCaseWriter、CalculiXRunner、CalculiXResultReader；
不要把具体 .inp 拼写细节全部写进插件管理器。
```

### 成员 B：CalculiX 输入文件导出

负责：

```text
MeshData / SimulationCase / Material / BoundaryCondition / Load
  -> CalculiX job.inp
```

主要文件：

```text
src/solver/calculix/CalculiXCaseWriter.*
src/solver/calculix/CalculiXInputDeck.*
src/mesh/MshReader.*
src/mesh/MeshData.h
src/mesh/MeshBoundary.h
src/project/MeshRepository.*
```

成员 B 的原则：

```text
只负责生成正确的 CalculiX 输入；
缺少网格、材料、约束、载荷时返回明确错误；
不要直接调用 ccx；
不要直接操作 UI。
```

### 成员 C：CalculiX 运行、结果和 UI

负责：

```text
job.inp
  -> ccx
  -> .sta / .dat / .frd
  -> ResultObject
  -> UI 展示
```

主要文件：

```text
src/solver/calculix/CalculiXRunner.*
src/solver/calculix/CalculiXResultReader.*
src/result/ResultObject.h
src/project/ResultRepository.*
src/ui/property/ResultPropertyView.*
src/ui/ProjectTreePanel.*
src/ui/PropertyPanel.*
项目功能测试流程.md
```

成员 C 的原则：

```text
只负责运行、结果摘要和展示；
不要修改 CalculiX .inp 生成逻辑；
不要绕过 SolverCaseWorkflowController 直接在 UI 里拼业务流程。
```

## 3. 冻结接口

第一阶段不要删除或重命名下面字段。

### 3.1 MeshBoundary

```cpp
QString id;
QString name;
QString meshName;
QString sourceGeometryName;
QString sourceFaceGroupId;
QString physicalGroupName;
int physicalGroupTag;
int faceCount;
```

### 3.2 ResultObject

```cpp
QString id;
QString name;
QString solverName;
QString casePath;
QString logFile;
QString createdAt;
bool success;
QString summary;
```

### 3.3 SolverCaseWriterResult

```cpp
bool success;
QString caseRootPath;
QStringList writtenFiles;
QStringList warnings;
QStringList errors;
QStringList logMessages;
```

后续可以新增：

```text
SolverRunResult
SolverResultReadResult
SolverPluginDescriptor
```

但不要破坏现有结果结构。

## 4. OpenFOAM 当前处理方式

OpenFOAM 现在只做接口保留。

允许存在：

```text
OpenFoamCaseWriter
OpenFoamPlugin placeholder
```

行为要求：

```text
1. 构建通过
2. 插件机制能表达 OpenFOAM
3. 调用时返回明确 not implemented
4. 不影响 CalculiX 导出和运行
```

不要在第一阶段投入大量时间写 OpenFOAM 的 `0/U`、`0/p`、`system`、`constant`。

## 5. 高冲突文件规则

下面文件不要三个人同时改：

```text
CMakeLists.txt
src/ui/MainWindow.cpp
src/ui/MainWindow.h
src/project/ProjectContext.*
src/project/ProjectModel.*
src/solver/SimulationCase.h
src/solver/BoundaryCondition.h
src/solver/Material.h
src/solver/Load.h
```

规则：

```text
1. CMakeLists.txt 由一个人集中合并新增文件
2. MainWindow 菜单/工具栏入口由成员 A 或 C 统一接
3. ProjectContext / ProjectModel 第一阶段尽量不再改
4. Material / BoundaryCondition / Load 不够用时，先用 adapter 或 extraProperties
```

## 6. 推荐并行顺序

### 第 1 轮

成员 A：

```text
升级 SolverPlugin 接口
新增 SolverRunResult
新增 SolverResultReadResult
新增 CalculiXPlugin 骨架
保持 OpenFOAM placeholder
```

成员 B：

```text
新增 CalculiXCaseWriter
输出 *NODE
输出 *ELEMENT, TYPE=C3D4
输出 *MATERIAL / *ELASTIC
```

成员 C：

```text
新增 CalculiXRunner
新增 CalculiXResultReader
新增 ResultPropertyView 草案
```

### 第 2 轮

成员 A：

```text
把 CalculiXCaseWriter / CalculiXRunner / CalculiXResultReader 串进 CalculiXPlugin
新增 SolverCaseWorkflowController
```

成员 B：

```text
补 *BOUNDARY
补 *CLOAD 或 *DLOAD
补错误诊断
```

成员 C：

```text
UI 接 Export / Run
运行后写入 ResultRepository
属性面板展示 Result
```

### 第 3 轮

成员 A：

```text
整理插件架构说明
说明 OpenFOAM 如何沿用同一接口扩展
```

成员 B：

```text
准备一个最小可运行 CalculiX 算例
```

成员 C：

```text
更新项目功能测试流程
整理演示步骤
```

## 7. 每个人完成定义

### 成员 A 完成定义

```text
1. SolverPlugin 能表达 export/run/readResult
2. CalculiXPlugin 可被统一调用
3. OpenFOAM placeholder 存在且不影响 CalculiX
4. SolverCaseWorkflowController 能串联导出、运行、读取结果
5. Debug 构建通过
```

### 成员 B 完成定义

```text
1. CalculiXCaseWriter 能生成 job.inp
2. job.inp 至少包含 *NODE / *ELEMENT / *MATERIAL / *ELASTIC
3. 能处理 tetra4 -> C3D4
4. 缺少必要数据时有明确错误
5. Debug 构建通过
```

### 成员 C 完成定义

```text
1. CalculiXRunner 能调用 ccx
2. CalculiXResultReader 能识别 .sta / .dat / .frd
3. 能生成 ResultObject
4. UI 能展示 Result 基本信息
5. 测试文档能复现流程
6. Debug 构建通过
```

## 8. 结论

现在三个人可以同时开始。

分工应调整为：

```text
你：成员 A，插件架构与集成
组员 1：成员 B，CalculiX 输入文件导出
组员 2：成员 C，CalculiX 运行、结果读取、UI 与验证
```

当前第一目标是：

```text
CalculiX 跑通；
OpenFOAM 接口留好；
插件机制讲清楚。
```
