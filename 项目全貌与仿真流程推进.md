# MyCAE 项目全貌与仿真流程推进方向

本文档描述当前 MyCAE 的代码架构、核心业务链路、已完成能力，以及下一阶段围绕“完整仿真流程”的推进方向。

## 1. 当前项目定位

MyCAE 当前已经从一个 Qt + OCC + VTK + Gmsh 的功能原型，逐步演进为轻量级 CAE 桌面软件框架。

当前目标不是只做几何显示或网格显示，而是打通下面这条完整链路：

```text
项目创建/打开
  -> 几何创建/导入/布尔
  -> 拾取面
  -> 创建 FaceGroup
  -> 绑定边界条件/材料/载荷
  -> 生成网格
  -> 导出求解器 case
  -> 运行求解器
  -> 读取结果/后处理显示
```

目前前半段已经比较清楚，后半段需要继续集中推进。

## 2. 当前代码分层

当前源码主要分为以下模块：

```text
src/
├─ commands/      QAction 到业务命令的薄适配层
├─ geometry/      几何数据、几何创建、FaceGroup 服务
├─ mesh/          Gmsh 调用、MSH 读取、网格对象、GmshCaseWriter 诊断
├─ occ/           Open CASCADE 几何构造、布尔、STEP/BREP、VTK 转换
├─ picking/       拾取模式、拾取集合、PickController
├─ project/       ProjectContext、Repository、SelectionState、项目加载
├─ render/        渲染高亮、VTK 拾取适配、面几何计算、highlight actor
├─ solver/        材料、边界条件、载荷、SimulationCase、求解器插件
├─ ui/            Qt 窗口、面板、对话框、ActionRegistry
└─ workflow/      项目/几何/网格/面组/选择/求解器流程编排
```

### 2.1 project 层

`ProjectModel` 已经被拆成 facade，真实数据由 `ProjectContext` 持有：

```text
ProjectContext
├─ Project
├─ GeometryRepository
├─ MeshRepository
├─ SolverRepository
└─ SelectionState
```

对应职责：

- `GeometryRepository`：管理 `GeometryObject`、`BoxGeometry`、`CylinderGeometry`
- `MeshRepository`：管理 `MeshObject`
- `SolverRepository`：管理 `Material`、`BoundaryCondition`、`Load`、`FaceGroup`
- `SelectionState`：统一当前选择和可用操作状态
- `ProjectModel`：保留旧接口，作为兼容 facade，减少一次性破坏调用点

这一步之后，各模块的数据边界已经比最初清楚很多。

### 2.2 workflow 层

当前 workflow 层已经承担主要流程编排：

- `ProjectWorkflowController`
  - 创建/打开项目
  - 加载几何、网格、simulation case
  - 刷新项目树
  - 保存 simulation case

- `GeometryWorkflowController`
  - 创建几何
  - 显示几何
  - 布尔几何

- `MeshWorkflowController`
  - 检查 Gmsh
  - 生成网格
  - 读取网格信息
  - 显示网格

- `FaceGroupWorkflowController`
  - 创建/替换面组
  - 添加/移除拾取面
  - 清空/重命名/删除面组
  - 设置局部网格尺寸
  - 设置 physical group 开关
  - 保存 case、刷新树、重新选择并高亮

- `SelectionController`
  - 统一处理树节点选择
  - 刷新属性面板
  - 触发几何显示/面组高亮/边界条件引用面组高亮

- `SolverWorkflowController`
  - 保存 simulation case
  - 检查外部求解器
  - 运行求解器插件

### 2.3 command 层

当前已经引入轻量 `ActionRegistry` / `AppCommand`。

方向是：

```text
QAction
  -> actionId
  -> ActionRegistry
  -> AppCommand
  -> WorkflowController / Service
```

也就是说，`QAction` 不再直接绑定大量 `MainWindow::xxxBusinessFunction`。

当前 command 已拆分为：

- `ProjectCommands`
- `GeometryCommands`
- `MeshCommands`
- `SolverCommands`
- `PickModeCommands`
- `FaceGroupEditCommands`
- `UtilityCommands`
- `CommandUtilities`
- `FaceGroupCommandUtilities`

### 2.4 picking / face group / selection

当前拾取系统的主链路是：

```text
VtkRenderCanvas
  -> VtkPickAdapter
  -> PickSelection
  -> PickController
  -> FaceGroupWorkflowController
  -> FaceGroupService
  -> SolverRepository.faceGroups
  -> RenderHighlightController
```

当前已经支持：

- 面拾取模式
- 拾取集合状态
- 当前拾取面高亮
- 根据拾取面创建 FaceGroup
- 添加拾取面到已有 FaceGroup
- 从 FaceGroup 移除拾取面
- 清空 FaceGroup
- 重命名 FaceGroup
- 删除 FaceGroup
- 删除时修复边界条件引用
- 选中 FaceGroup 自动高亮
- 选中 BoundaryCondition 自动高亮它引用的 FaceGroup
- 属性面板显示 PickMode、几何名、面数量、face indices

正确的数据关系是：

```text
PickSelection
  -> FaceGroup
  -> BoundaryCondition
  -> Material / Load / Solver export

FaceGroup
  -> LocalMeshRule / PhysicalGroup
```

不要让“拾取面”直接绑定材料或物理约束。拾取面只是临时 UI 状态，FaceGroup 才是稳定业务对象。

### 2.5 render 层

VTK 相关逻辑已经从 `VtkRenderCanvas` 拆出：

- `VtkPickAdapter`：负责 VTK cell picker 到 `PickSelection`
- `VtkFaceGeometry`：负责面中心、法向、面积等几何信息
- `VtkHighlightActorFactory`：负责构造高亮 actor
- `RenderHighlightController`：负责按业务对象触发高亮

`VtkRenderCanvas` 现在主要是 Qt/VTK widget 外壳和 scene 状态管理。

### 2.6 solver 层

当前 solver 数据结构包括：

- `Material`
- `BoundaryCondition`
- `Load`
- `SimulationCase`
- `SimulationCaseBuilder`
- `SimulationCaseJsonReader`
- `SimulationCaseJsonWriter`
- `SimulationCaseManager`
- `SolverDataService`
- `SolverPluginManager`
- `ExternalProcessSolverPlugin`

目前已经做到：

- 材料创建/编辑/删除
- 边界条件创建/编辑/删除
- 载荷创建/编辑/删除
- 边界条件可引用 FaceGroup
- 边界条件可引用 Material
- Load 可引用 BoundaryCondition
- 保存/读取 simulation case JSON
- 外部求解器插件框架

但真正完整仿真流程还缺关键导出和结果回读。

## 3. 当前完整仿真流程成熟度

### 3.1 已经比较稳定的部分

```text
项目管理
  -> 几何管理
  -> 拾取面
  -> FaceGroup
  -> 材料/边界条件/载荷数据管理
  -> simulation case JSON 保存
```

这些已经具备可继续扩展的架构基础。

### 3.2 仍是半成品的部分

```text
FaceGroup
  -> Gmsh physical group / local mesh
```

当前已经有：

- `FaceGroup.faceReferences`
- `physicalGroupEnabled`
- `localMeshEnabled`
- `localMeshSize`
- `GmshCaseWriter`

但 `GmshCaseWriter` 目前只做诊断，不会伪造 Gmsh physical group。

原因是：

```text
OCC face index
  != 必然稳定等于
Gmsh surface tag
```

如果现在硬写 physical group，很可能生成一个“看起来成功但边界错位”的网格，这是 CAE 软件里非常危险的错误。

### 3.3 还没有打通的部分

```text
SimulationCase
  -> 求解器 case 文件夹
  -> 求解器运行
  -> 结果文件
  -> VTK/ParaView 后处理
```

这是下一阶段最值得集中推进的主线。

## 4. 下一步推进方向：完整仿真流程优先

你现在偏向继续推进完整仿真流程，这个判断是对的。

下一步不建议继续泛泛地拆 UI，而应该围绕一条最小但真实的仿真闭环推进：

```text
几何
  -> 面组
  -> 边界条件
  -> 材料
  -> 网格
  -> 求解器 case
  -> 运行
  -> 结果读取
```

## 5. 推荐优先级

### P1：新增 SolverCaseWriter

目标：

把当前 `SimulationCase` 转换为真实求解器文件夹，而不是只保存 JSON。

建议新增：

```text
src/solver/export/
├─ SolverCaseWriter.h
├─ SolverCaseWriter.cpp
├─ SolverCaseWriterResult.h
├─ OpenFoamCaseWriter.h
├─ OpenFoamCaseWriter.cpp
└─ OpenFoamBoundaryMapper.h
```

先不要做大而全，先支持一个最小 OpenFOAM case：

```text
case/
├─ 0/
│  ├─ U
│  ├─ p
│  ├─ k
│  ├─ omega
│  └─ nut
├─ constant/
│  ├─ transportProperties
│  ├─ turbulenceProperties
│  └─ polyMesh/
└─ system/
   ├─ controlDict
   ├─ fvSchemes
   └─ fvSolution
```

第一阶段可以先不处理复杂物理，只支持：

- steady incompressible
- SIMPLE
- velocity inlet
- pressure outlet
- wall
- symmetry
- Newtonian fluid

### P2：明确 Mesh -> Boundary 的映射策略

完整仿真最关键的问题是：

```text
FaceGroup
  -> Gmsh physical group
  -> mesh boundary patch
  -> OpenFOAM boundary
```

当前最可靠推进方式：

1. 先让 `GmshCaseWriter` 生成 `.geo` 或 `.geo_unrolled`
2. 在 `.geo` 里导入 STEP
3. 尝试建立 OCC face index 到 Gmsh surface tag 的映射
4. 成功时写：

```text
Physical Surface("inlet") = {...};
Physical Surface("wall") = {...};
```

5. 失败时阻止导出仿真 case，并给明确错误日志

这里不能只追求能跑，必须保证边界条件没有错绑。

### P3：新增 MeshBoundaryRepository 或 MeshBoundary 数据结构

现在 `BoundaryCondition` 引用的是：

```text
BoundaryTarget
├─ geometryName
├─ faceGroupId
├─ faceGroupName
└─ meshBoundaryName
```

下一步应该补一个明确的网格边界对象：

```cpp
struct MeshBoundary {
    QString name;
    QString sourceGeometryName;
    QString sourceFaceGroupId;
    QString physicalGroupName;
    int physicalGroupTag = -1;
    int faceCount = 0;
};
```

这样链路会变成：

```text
FaceGroup
  -> MeshBoundary
  -> BoundaryCondition
  -> SolverCaseWriter
```

这比让 `BoundaryCondition` 直接猜 mesh patch 稳定。

### P4：实现 OpenFOAM 最小导出闭环

建议第一版只做最小可验证 case：

```text
一个几何
一个网格
一个 inlet FaceGroup
一个 outlet FaceGroup
一个 wall/default FaceGroup
一个流体材料
一个 SIMPLE 求解器配置
```

导出目标：

```text
project_root/
└─ solver/
   └─ openfoam/
      └─ caseName/
```

需要生成：

- `0/U`
- `0/p`
- `constant/transportProperties`
- `constant/turbulenceProperties`
- `system/controlDict`
- `system/fvSchemes`
- `system/fvSolution`

如果 mesh 已经由 Gmsh 生成，还需要：

- 调用 `gmshToFoam`
- 修正 boundary patch type
- 根据 `BoundaryCondition` 写入 `U/p` 字段边界

### P5：求解器运行流程规范化

当前已有 solver plugin 思路，下一步应该把运行流程明确成：

```text
SolverCaseWriter::write()
  -> SolverPluginManager::plugin()
  -> SolverPlugin::check()
  -> SolverPlugin::run()
  -> SolverRunResult
  -> LogPanel
  -> ResultRepository
```

建议新增：

```text
SolverRunController
SolverCaseWorkflowController
ResultRepository
ResultObject
```

其中 `SolverWorkflowController` 可以继续作为 facade。

### P6：结果读取和后处理

初期不要急着自己做复杂 CFD 后处理。

推荐先支持：

- 打开 ParaView
- 记录结果目录
- 检测 OpenFOAM 时间步目录
- 显示 residual/log 摘要

再往后才做：

- 读取 VTK/foam 转换结果
- 在 VTK 里显示速度云图/压力云图
- 流线、切片、等值面

## 7. 当前最建议马上做的事

下一步最建议直接做：

```text
新增 SolverCaseWriter / OpenFoamCaseWriter
```

原因：

- 它直接服务完整仿真流程
- 不依赖 UI 大改
- 能把材料、边界条件、载荷这些已有数据真正用起来
- 能暴露当前数据结构还缺什么
- 比继续抽象架构更接近 CAE 软件核心价值

第一版不要追求完美求解，只追求“导出结构正确、日志明确、失败原因明确”。

## 8. 当前风险点

### 8.1 FaceGroup 拓扑稳定性

`faceIndex` 当前来自 OCC/VTK 面遍历顺序。几何重新生成或布尔操作后，face index 可能变化。

已有缓解基础：

- `FaceReference.center`
- `FaceReference.normal`
- `FaceReference.area`
- `pickedPoint`

后续应加入 fallback matching。

### 8.2 Gmsh surface tag 映射

这是完整仿真流程最大技术风险。

必须把映射显式建模，不能假设：

```text
OCC face index == Gmsh surface tag
```

### 8.3 SolverDataController 仍偏 UI

虽然已有 `SolverDataService`，但 `SolverDataController` 仍承担不少对话框和属性刷新逻辑。

这不是当前第一优先级。完整仿真流程跑通后再继续拆更合适。

### 8.4 MainWindow 仍偏重

`MainWindow` 已经比最初轻很多，但仍有 500 多行。

现在不建议优先继续拆它。只要新增功能继续走 command/workflow/service，`MainWindow` 会自然稳定下来。

## 9. 结论

当前 MyCAE 的架构已经进入“可以承载完整仿真流程”的阶段。

下一步主线应该从架构拆分转向仿真闭环：

```text
SimulationCase
  -> SolverCaseWriter
  -> MeshBoundary
  -> Gmsh physical group
  -> OpenFOAM case
  -> Solver run
  -> Result management
```

最推荐马上开始的是：

```text
阶段 1：实现 SolverCaseWriter / OpenFoamCaseWriter 骨架
```

它会成为后续完整仿真流程的核心出口。
