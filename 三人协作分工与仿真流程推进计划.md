# MyCAE 三人协作分工与仿真流程推进计划

本文档是当前阶段的协作基准。当前目标调整为：

```text
主线：接入 CalculiX
辅线：预留 OpenFOAM 接口
核心：形成标准化 SolverPlugin 插件机制
```

也就是说，现阶段不是同时完整实现 CalculiX 和 OpenFOAM，而是：

```text
用 CalculiX 验证完整求解器插件闭环；
用 OpenFOAM 接口证明架构具备继续扩展主流 CFD 求解器的能力。
```

## 1. 老师任务对应关系

老师要求的必做项：

```text
设计标准化的求解器接口和插件化机制，
让求解器集成变得简单，
形成开放架构，
可以无缝对接主流及自研求解器。
```

我们选择的落地方向：

```text
（2）设计自研求解器快速封装机制
```

理由：

- CalculiX 是外部结构有限元求解器，不是典型 CFD 求解器；
- 它适合证明“任意外部求解器都可以通过标准接口接入”；
- 后续 OpenFOAM、SU2、自研 C++/Fortran/Python 求解器都可以复用同一套机制；
- OpenFOAM 暂时只保留接口和 placeholder，不作为第一阶段完整实现目标。

对外表述建议：

```text
本项目设计了标准化 SolverPlugin 插件接口，将求解器集成拆分为 CaseWriter、Runner、ResultReader 三个阶段。
当前以 CalculiX 作为第一个外部求解器插件完成验证，并为 OpenFOAM 预留同构插件接口。
该机制后续可扩展到 OpenFOAM、SU2、Fluent、CFX、FoamExtend 或自研 C++/Fortran/Python 求解器。
```

## 2. 当前总体目标

第一阶段目标是跑通 CalculiX 最小结构仿真闭环：

```text
项目
  -> 几何
  -> 网格
  -> FaceGroup / MeshBoundary
  -> 材料
  -> 约束 / 载荷
  -> CalculiX job.inp
  -> ccx 运行
  -> .dat / .sta / .frd
  -> ResultObject
```

OpenFOAM 当前只做接口保留：

```text
SolverPlugin
  -> OpenFoamCaseWriter placeholder
  -> 后续可扩展 OpenFOAM case 导出
```

## 3. 三人新分工

推荐三个人按下面方式并行：

```text
成员 A：插件架构与集成主线
成员 B：CalculiX 输入文件导出主线
成员 C：CalculiX 运行、结果读取、UI 与验证主线
```

你建议负责成员 A，因为这条线控制全局接口和最终集成。

## 4. 成员 A：插件架构与集成主线

### 4.1 主要职责

成员 A 负责统一标准，不直接陷入所有 CalculiX 细节。

核心职责：

```text
1. 定义标准化 SolverPlugin 接口
2. 定义 CaseWriter / Runner / ResultReader 三段式机制
3. 定义 SolverPluginDescriptor
4. 统一 SolverCaseWriterResult / SolverRunResult / SolverResultReadResult
5. 维护 SolverPluginManager
6. 接入 CalculiXPlugin
7. 保留 OpenFOAMPlugin / OpenFoamCaseWriter 接口
8. 做最终集成验收
```

### 4.2 建议负责文件

```text
src/solver/plugin/SolverPlugin.h
src/solver/plugin/SolverPluginManager.*
src/solver/export/SolverCaseWriter.*
src/solver/export/SolverCaseWriterResult.h
src/solver/export/OpenFoamCaseWriter.*
src/solver/calculix/CalculiXPlugin.*
src/workflow/SolverCaseWorkflowController.*
```

### 4.3 第一阶段要做

先把求解器插件接口从“简单外部脚本”升级成标准生命周期：

```text
exportCase
runCase
readResult
```

每个阶段都要有结构化结果：

```cpp
struct SolverCaseWriterResult {
    bool success = false;
    QString caseRootPath;
    QStringList writtenFiles;
    QStringList warnings;
    QStringList errors;
    QStringList logMessages;
};

struct SolverRunResult {
    bool success = false;
    int exitCode = -1;
    QString command;
    QString workingDirectory;
    QString standardOutput;
    QString standardError;
    QString logFile;
    QStringList errors;
    QStringList logMessages;
};

struct SolverResultReadResult {
    bool success = false;
    QString summary;
    QStringList resultFiles;
    QStringList warnings;
    QStringList errors;
    QStringList logMessages;
};
```

### 4.4 OpenFOAM 如何保留

OpenFOAM 当前只需要保留：

```text
OpenFoamCaseWriter.h/.cpp
OpenFoamPlugin.h/.cpp 或 placeholder
```

行为：

```text
接口存在；
能在插件列表中说明 planned / not implemented；
调用时返回明确错误：OpenFOAM export is reserved but not implemented.
```

不要现在投入大量时间写 OpenFOAM case。

### 4.5 成员 A 验收标准

```text
1. SolverPlugin 接口能同时表达 CalculiX 和 OpenFOAM
2. CalculiXPlugin 能被 SolverPluginManager 注册或创建
3. SolverCaseWorkflowController 能统一调用 export/run/readResult
4. OpenFOAM placeholder 不影响 CalculiX 使用
5. Debug 构建通过
```

## 5. 成员 B：CalculiX 输入文件导出主线

### 5.1 主要职责

成员 B 负责把 MyCAE 的项目数据转换成 CalculiX `.inp`。

核心链路：

```text
ProjectModel
  -> SimulationCase
  -> MeshObject / MeshData
  -> Material
  -> BoundaryCondition
  -> Load
  -> CalculiX job.inp
```

### 5.2 建议负责文件

```text
src/solver/calculix/CalculiXCaseWriter.h
src/solver/calculix/CalculiXCaseWriter.cpp
src/solver/calculix/CalculiXInputDeck.h
src/solver/calculix/CalculiXInputDeck.cpp
src/mesh/MshReader.*
src/mesh/MeshData.h
src/mesh/MeshBoundary.h
src/project/MeshRepository.*
```

### 5.3 第一阶段要做

先导出最小可运行 `.inp`：

```text
*HEADING
*NODE
*ELEMENT, TYPE=C3D4
*MATERIAL
*ELASTIC
*SOLID SECTION
*BOUNDARY
*CLOAD 或 *DLOAD
*STEP
*STATIC
*END STEP
```

### 5.4 数据映射规则

建议先按下面规则做：

```text
MeshData.nodes
  -> *NODE

MeshData.tetrahedra
  -> *ELEMENT, TYPE=C3D4

Material.extraProperties["youngModulus"]
  -> *ELASTIC 第一列

Material.extraProperties["poissonRatio"]
  -> *ELASTIC 第二列

BoundaryCondition(type=Wall 或 Unknown, target=FaceGroup/MeshBoundary)
  -> *BOUNDARY

Load(type=Force/Pressure 暂用现有 LoadType 映射)
  -> *CLOAD 或 *DLOAD
```

如果当前材料结构不够，不要大改 `Material`，先使用：

```text
Material.extraProperties
```

例如：

```text
youngModulus = 2.1e11 Pa
poissonRatio = 0.3
density = 7800 kg/m^3
```

### 5.5 FaceGroup / MeshBoundary 映射

CalculiX 需要集合：

```text
*NSET
*ELSET
*SURFACE
```

第一阶段可以先做保守实现：

```text
MeshBoundary
  -> NSET / ELSET / SURFACE
```

如果 MeshBoundary 还不稳定，则导出失败并提示：

```text
CalculiX export failed: no mesh boundary is available for selected boundary condition.
```

不要静默生成错误集合。

### 5.6 成员 B 验收标准

```text
1. 能从 MSH 或 MeshData 导出 *NODE / *ELEMENT
2. 能生成 job.inp
3. 缺少网格、材料、约束、载荷时有明确错误
4. 至少支持 tetra4 -> C3D4
5. Debug 构建通过
```

## 6. 成员 C：CalculiX 运行、结果读取、UI 与验证

### 6.1 主要职责

成员 C 负责让 CalculiX 能被实际运行，并把结果纳入 MyCAE。

核心链路：

```text
job.inp
  -> ccx job
  -> job.sta / job.dat / job.frd
  -> ResultObject
  -> ProjectTreePanel / PropertyPanel
```

### 6.2 建议负责文件

```text
src/solver/calculix/CalculiXRunner.h
src/solver/calculix/CalculiXRunner.cpp
src/solver/calculix/CalculiXResultReader.h
src/solver/calculix/CalculiXResultReader.cpp
src/result/ResultObject.h
src/project/ResultRepository.*
src/ui/property/ResultPropertyView.*
src/ui/ProjectTreePanel.*
src/ui/PropertyPanel.*
项目功能测试流程.md
```

### 6.3 第一阶段要做

实现运行和结果摘要：

```text
1. 找到 ccx.exe
2. 在 caseDirectory 下运行 ccx job
3. 捕获 stdout/stderr
4. 检查 job.sta / job.dat / job.frd 是否存在
5. 读取 .sta 判断运行是否完成
6. 生成 ResultObject
```

### 6.4 ResultObject 使用

当前已预留：

```cpp
struct ResultObject
{
    QString id;
    QString name;
    QString solverName;
    QString casePath;
    QString logFile;
    QString createdAt;
    bool success = false;
    QString summary;
};
```

成员 C 可以在运行完成后写入：

```text
ProjectModel.resultRepository().results()
```

### 6.5 UI 最小入口

第一阶段 UI 不要做复杂。

只需要：

```text
Select Solver: CalculiX
Export Solver Case
Run Solver Case
Open Result Folder
```

OpenFOAM 可以出现在列表中，但显示：

```text
Reserved / Not implemented
```

### 6.6 成员 C 验收标准

```text
1. 能调用 ccx 运行 job.inp
2. 能识别 job.sta / job.dat / job.frd
3. 能生成 ResultObject
4. 项目树或属性面板能显示 Result 基本信息
5. 测试文档能指导别人复现流程
6. Debug 构建通过
```

## 7. 三人接口边界

### 7.1 成员 A 给 B/C 的接口

成员 A 需要尽早稳定：

```text
SolverCaseWriterResult
SolverRunResult
SolverResultReadResult
SolverPlugin
SolverPluginManager
SolverCaseWorkflowController
```

B/C 不要绕过这些接口直接从 UI 调 CalculiX。

### 7.2 成员 B 给 A/C 的接口

成员 B 需要提供：

```text
CalculiXCaseWriter::write(...)
生成的 job.inp 路径
writtenFiles 列表
错误信息
```

A 用它接入插件生命周期。

C 用它判断能否运行。

### 7.3 成员 C 给 A/B 的接口

成员 C 需要提供：

```text
CalculiXRunner::run(...)
CalculiXResultReader::read(...)
ResultObject 生成规则
UI 触发入口
```

A 用它完成插件闭环。

B 用它验证导出的 `.inp` 是否真的能运行。

## 8. 当前不要做的事

为了保证三人能并行，当前不要做：

```text
1. 不要完整实现 OpenFOAM case writer
2. 不要大改 Material / BoundaryCondition / Load 数据结构
3. 不要多人同时改 MainWindow
4. 不要多人同时改 ProjectContext / ProjectModel
5. 不要直接把 CalculiX 逻辑写进 UI
```

如果现有结构不够，优先通过 adapter / mapper 解决。
