# MyCAE 三人协作分工与仿真流程推进计划

本文档用于把当前 MyCAE 的后续开发拆成三个可以并行推进的工作包。目标不是继续零散加功能，而是围绕“完整仿真流程”形成清晰协作边界。

## 1. 总目标

当前阶段的总目标是打通第一条完整仿真闭环：

```text
几何创建/导入
  -> 面拾取
  -> FaceGroup
  -> 边界条件/材料/载荷
  -> 网格生成
  -> 求解器 case 导出
  -> 求解器运行
  -> 结果记录/查看
```

下一阶段优先完成的是：

```text
SimulationCase
  -> SolverCaseWriter
  -> MeshBoundary
  -> Gmsh physical group
  -> OpenFOAM case
  -> solver run
  -> result management
```

## 2. 三人工作包划分

建议三个人分别负责：

```text
成员 A：Solver Case 导出与求解器运行主线
成员 B：FaceGroup / MeshBoundary / Gmsh physical group 主线
成员 C：UI 工作流、项目树、结果管理、验证文档主线
```

这三个方向互相依赖，但可以通过明确接口并行开发。

## 3. 总体架构补全方向

当前项目已有基础：

```text
ProjectContext
├─ Project
├─ GeometryRepository
├─ MeshRepository
├─ SolverRepository
└─ SelectionState
```

下一步建议补全：

```text
SolverRepository
├─ materials
├─ boundaryConditions
├─ loads
└─ faceGroups

MeshRepository
├─ meshObjects
└─ meshBoundaries

ResultRepository
└─ resultObjects
```

新增核心流程：

```text
SimulationCaseBuilder
  -> SolverCaseWriter
  -> OpenFoamCaseWriter
  -> SolverPlugin
  -> ResultRepository
```

新增网格边界链路：

```text
FaceGroup
  -> GmshCaseWriter
  -> MeshBoundary
  -> BoundaryCondition
  -> OpenFoamCaseWriter
```

## 4. 统一接口约定

为了三人并行，先约定几个核心数据结构。

### 4.1 MeshBoundary

建议新增：

```cpp
struct MeshBoundary
{
    QString id;
    QString name;
    QString meshName;
    QString sourceGeometryName;
    QString sourceFaceGroupId;
    QString physicalGroupName;
    int physicalGroupTag = -1;
    int faceCount = 0;
};
```

作用：

- 记录网格里的真实边界 patch
- 连接 `FaceGroup` 和 `BoundaryCondition`
- 避免边界条件直接猜测 Gmsh/OpenFOAM patch 名称

### 4.2 SolverCaseWriterResult

建议新增：

```cpp
struct SolverCaseWriterResult
{
    bool success = false;
    QString caseRootPath;
    QStringList writtenFiles;
    QStringList warnings;
    QStringList errors;
    QStringList logMessages;
};
```

作用：

- 明确 case 是否导出成功
- 明确生成了哪些文件
- 明确哪些问题是 warning，哪些是 error

### 4.3 ResultObject

建议新增：

```cpp
struct ResultObject
{
    QString id;
    QString name;
    QString solverName;
    QString casePath;
    QString logFile;
    QString createdAt;
    bool success = false;
    QString summary;
};
```

作用：

- 记录一次求解结果
- 项目树可以显示 Result
- 属性面板可以显示结果路径、日志、状态

## 5. 成员 A：Solver Case 导出与求解器运行

### 5.1 主要目标

负责把当前 `SimulationCase` 转成真实求解器 case，并接入求解器运行流程。

主线：

```text
SimulationCase
  -> SolverCaseWriter
  -> OpenFoamCaseWriter
  -> SolverPlugin
  -> SolverRunResult
```

### 5.2 建议新增文件

```text
src/solver/export/SolverCaseWriter.h
src/solver/export/SolverCaseWriter.cpp
src/solver/export/OpenFoamCaseWriter.h
src/solver/export/OpenFoamCaseWriter.cpp
src/solver/export/SolverCaseWriterResult.h
src/workflow/SolverCaseWorkflowController.h
src/workflow/SolverCaseWorkflowController.cpp
```

### 5.3 第一阶段任务

先实现最小 OpenFOAM case 导出，不运行。

需要生成：

```text
solver/openfoam/<caseName>/
├─ 0/
│  ├─ U
│  └─ p
├─ constant/
│  ├─ transportProperties
│  └─ turbulenceProperties
└─ system/
   ├─ controlDict
   ├─ fvSchemes
   └─ fvSolution
```

第一阶段暂时允许：

- 只支持 incompressible SIMPLE
- 只支持 Newtonian fluid
- 只支持 `U`、`p`
- 先不处理复杂湍流文件
- 没有有效 MeshBoundary 时拒绝导出并给错误日志

### 5.4 第二阶段任务

接入求解器运行。

需要完成：

- `SolverCaseWorkflowController::exportCase`
- `SolverCaseWorkflowController::runCase`
- 调用已有 `SolverPluginManager`
- 记录 `SolverRunResult`
- 把运行日志送到 `LogPanel`

### 5.5 验收标准

至少满足：

```text
1. 能从当前 ProjectModel 构建 SimulationCase
2. 能导出 OpenFOAM case 文件夹
3. 导出的 U/p/controlDict/fvSchemes/fvSolution 文件存在
4. 缺少材料、边界、网格时给明确错误
5. Debug 构建通过
```

### 5.6 注意事项

不要把导出逻辑写进 `MainWindow`。

正确调用链：

```text
Command
  -> SolverCaseWorkflowController
  -> SolverCaseWriter
  -> OpenFoamCaseWriter
```

## 6. 成员 B：FaceGroup / MeshBoundary / Gmsh physical group

### 6.1 主要目标

负责把拾取的 `FaceGroup` 变成真实网格边界，使边界条件能可靠绑定到网格 patch。

主线：

```text
FaceGroup
  -> Gmsh physical group
  -> MeshBoundary
  -> BoundaryCondition
```

### 6.2 建议新增或修改文件

```text
src/mesh/MeshBoundary.h
src/project/MeshBoundaryRepository.h
src/project/MeshBoundaryRepository.cpp
src/mesh/GmshCaseWriter.h
src/mesh/GmshCaseWriter.cpp
src/mesh/MshReader.h
src/mesh/MshReader.cpp
src/workflow/MeshWorkflowController.cpp
```

如果不新增单独 `MeshBoundaryRepository`，也可以先把 `meshBoundaries` 放进 `MeshRepository`。

推荐结构：

```text
MeshRepository
├─ meshObjects
└─ meshBoundaries
```

### 6.3 第一阶段任务

先新增 `MeshBoundary` 数据结构和 repository 接口。

需要完成：

- `MeshRepository` 增加 `meshBoundaries`
- 支持 clear / find / append / replace
- `ProjectTreePanel` 后续可以显示 MeshBoundary，但第一阶段可先不做 UI

### 6.4 第二阶段任务

完善 `GmshCaseWriter`。

目标：

- 读取当前几何的 FaceGroup
- 判断哪些 FaceGroup 开启了 `physicalGroupEnabled`
- 尝试建立 OCC face index 到 Gmsh surface tag 的映射
- 成功时生成 physical group
- 失败时明确报错，不生成错误边界

重要原则：

```text
不能假设 OCC face index == Gmsh surface tag
```

### 6.5 第三阶段任务

扩展 `MshReader`，读取 physical names / boundary 信息。

需要得到：

```text
physicalGroupName
physicalGroupTag
faceCount
sourceFaceGroupId
```

然后写入：

```text
MeshRepository.meshBoundaries
```

### 6.6 验收标准

至少满足：

```text
1. FaceGroup 开启 physicalGroupEnabled 后，网格生成流程会尝试导出 physical group
2. 映射失败时有明确日志
3. 映射成功时 MSH 中能读出 physical group
4. MeshBoundary 能保存到项目内存模型
5. Debug 构建通过
```

### 6.7 注意事项

这是完整仿真流程里最容易出错的地方。

宁可暂时阻止导出，也不要静默生成错误边界。

### 7.3 第一阶段任务

- `ResultObject`
- `ResultRepository`
- 项目树新增 Results 节点

新增命令入口。

建议新增菜单或工具栏 action：


```text

```

### 7.5 第三阶段任务

维护验证文档。

需要更新：

```text
项目功能测试流程.md
```

新增完整流程测试：

```text
1. 新建项目
2. 创建几何
3. 进入面拾取
4. 创建 inlet/outlet/wall FaceGroup
5. 创建材料
6. 创建边界条件
7. 生成网格
8. 导出 solver case
9. 运行 solver
10. 查看 Result
```

### 7.6 验收标准

至少满足：

```text
1. 用户能从 UI 触发导出 case
2. 用户能从 UI 触发运行 solver
3. 运行后项目树出现 Result
4. 选中 Result 能看到路径和状态
5. 测试文档能指导别人完整验证流程
6. Debug 构建通过
```

## 8. 三人之间的接口边界

### 8.1 成员 A 需要成员 B 提供

```text
MeshBoundary 列表
BoundaryCondition 到 MeshBoundary 的可靠映射
physicalGroupName / patchName
```

如果 B 尚未完成，A 的 `OpenFoamCaseWriter` 应该明确失败：

```text
Export failed: no mesh boundaries are available. Generate mesh with physical groups first.
```

### 8.2 成员 B 需要成员 A 提供

```text
SolverCaseWriter 对 MeshBoundary 的字段需求
OpenFOAM patch name 规则
哪些 BoundaryConditionType 需要哪些 patch type
```

建议 patch name 使用：

```text
FaceGroup.name
```

并进行安全化：

```text
inlet  -> inlet
Outlet 1 -> Outlet_1
wall.left -> wall_left
```

### 8.3 成员 C 需要成员 A/B 提供

```text
WorkflowController 的公开方法
Result / MeshBoundary 数据结构
错误日志和状态字段
```

C 不应该直接拼业务逻辑，只负责 UI 触发和展示。

## 9. 推荐开发顺序

### 第 1 周：骨架并行

成员 A：

- 新增 `SolverCaseWriter`
- 新增 `OpenFoamCaseWriter`
- 生成最小 OpenFOAM 文件夹

成员 B：

- 新增 `MeshBoundary`
- 扩展 `MeshRepository`
- 梳理 `GmshCaseWriter` 输入输出

成员 C：

- 新增 Result 数据结构草案
- 设计项目树 Results 节点
- 更新测试流程文档草案

### 第 2 周：接口对接

成员 A：

- `OpenFoamCaseWriter` 使用 MeshBoundary 写 U/p boundaryField
- 缺少 MeshBoundary 时给明确错误

成员 B：

- Gmsh physical group 写入/诊断
- MSH physical name 读取

成员 C：

- UI 接入 Export Solver Case
- 属性面板显示 MeshBoundary / Result

### 第 3 周：最小闭环

成员 A：

- 接入 SolverPlugin 运行
- 生成运行日志

成员 B：

- 保证至少一个简单几何能生成可识别边界

成员 C：

- 项目树显示 Result
- 完整测试文档
- 手动走通一次流程

## 10. 合并策略

每个人尽量在自己的模块内改：

```text
成员 A：src/solver/export, src/workflow/SolverCaseWorkflowController.*
成员 B：src/mesh, src/project/MeshRepository.*, src/geometry/FaceGroup*
成员 C：src/ui, src/commands, src/result, docs
```

公共文件需要提前沟通：

```text
CMakeLists.txt
ProjectModel.h / ProjectContext.h
ProjectTreePanel.*
MainWindow.*
```

提交前必须：

```text
cmake --build build-codex-qt663-ninja --config Debug
```

## 11. 当前最小可交付版本定义

三人协作的第一版不要追求所有功能完美。

第一版只要求：

```text
1. 用户能创建几何
2. 用户能创建 FaceGroup
3. 用户能创建材料和边界条件
4. 用户能生成网格
5. 软件能导出 OpenFOAM case 文件夹
6. 缺少真实边界映射时给明确错误
7. 如果边界映射成功，可以运行求解器
8. 运行后能产生 Result 记录
```

## 12. 你本人建议负责的部分

如果你想控制项目主线，建议你负责成员 A：

```text
Solver Case 导出与求解器运行主线
```

原因：

- 这是完整仿真流程的出口
- 能倒逼材料、边界条件、网格边界这些结构变得合理
- 最贴近 CAE 软件最终价值
- 最适合你把 Foam-Agent / OpenFOAM 的理解迁移到 MyCAE

另外两个组员可以分别做：

- 组员 1：FaceGroup / MeshBoundary / Gmsh physical group
- 组员 2：UI 工作流 / Result / 测试文档

这样你可以把控核心仿真流程，组员分别补齐网格边界和使用体验。
